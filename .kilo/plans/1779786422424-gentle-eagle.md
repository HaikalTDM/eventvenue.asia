# Data Cleansing & Augmentation Plan

## Goal
Replace the current placeholder seed data (10 vendor users named "Vendor 1-10", 8 venues with generic descriptions, 3 services) with realistic production-grade synthetic data while retaining exactly **3 tagged mock records** — one per primary entity type — for demo/reference purposes.

---

## 1. Mock Record Retention Strategy

### 1.1 Which 4 Records Stay

| # | Entity Type | Record | Why |
|---|-------------|--------|-----|
| 1 | **Admin** | User `admin@eventvenue.asia` (role=admin) | UAT_PLAN §1.3.1 — hardcoded in frontend admin login |
| 2 | **Vendor (Venue Owner)** | User `aisha@majestic-kl.com` + VendorProfile "The Majestic KL" + Listing "Grand Ballroom at The Majestic Kuala Lumpur" | UAT_PLAN §1.2 — demo vendor account, featured venue on homepage |
| 3 | **Vendor (Service Provider)** | User `info@hassancatering.my` + VendorProfile "Hassan Premium Catering" + Listing "Hassan Premium Catering" (service) | UAT_PLAN §4.2 — demo service provider, referenced in bookings flow |
| 4 | **Customer** | User `customer@example.com` (role=customer, name="Demo Customer") | Provides a pre-authenticated customer session for demo/testing without sign-up |

These 3 are the ones referenced in the UAT plan and the frontend demo login buttons. Changing their emails/IDs breaks UAT.

### 1.2 Mock Tag Mechanism

Add an `is_mock` boolean column (default `false`) to three tables:

```
users.is_mock          BOOLEAN DEFAULT false
vendor_profiles.is_mock BOOLEAN DEFAULT false  
listings.is_mock       BOOLEAN DEFAULT false
```

This is a **non-breaking schema addition** — existing code ignores it, it doesn't affect queries, and it's trivially filterable (`WHERE is_mock = false` to exclude demo data in production queries).

## 3. Seed Script Rewrite Strategy

### 3.1 Atomic Approach

The new seed script (`lib/db/seed.ts`) runs in a single transaction:

1. **TRUNCATE** all tables (CASCADE) to start fresh
2. **INSERT** lookup tables (amenities, event_types)
3. **INSERT** 4 mock records with `is_mock = true`:
   - Admin user
   - Venue owner user + vendor profile + "Grand Ballroom" venue listing
   - Service provider user + vendor profile + "Hassan Premium Catering" service listing
   - Customer user
4. **INSERT** all synthetic data with `is_mock = false`
5. Run `drizzle-kit push` first to ensure `is_mock` columns exist

### 3.2 Idempotent Design

The seed script is designed to be re-runnable:
- TRUNCATE CASCADE clears all data
- No UUID hardcoding — all IDs are auto-generated
- Mock records are identified by their well-known emails, not IDs
- Can run `npx tsx lib/db/seed.ts` at any time to reset the database

### 3.3 File Changes

| File | Action |
|------|--------|
| `lib/db/schema/index.ts` | Add `is_mock` boolean column to `users`, `vendor_profiles`, `listings` |
| `lib/db/seed.ts` | Full rewrite with Faker.js generation |
| `lib/db/seed-helpers.ts` | NEW — helper functions for generating names, venues, reviews, etc. |
| `package.json` | Add `@faker-js/faker` as devDependency |

### 3.4 No Frontend Changes Required

- The `is_mock` column is purely a database-level tag. No API routes or frontend components need changes unless we later want to filter mock data from listings/search results. By default, mock records behave exactly like real records — they just have a flag for identification.

## 5. Risk & Considerations

- **Foreign key ordering**: Seed must insert users → vendor_profiles → listings → (photos, amenities, packages, blocks) → inquiries → bookings → reviews → favorites → conversations → messages. TRUNCATE CASCADE handles this cleanly.
- **Unique constraints**: Emails must be unique (Faker handles this). Customer-listing review pairs must be unique (track with Set).
- **Rating trigger**: The existing `trg_update_listing_rating` trigger auto-updates `listings.average_rating` and `review_count` when reviews are inserted. No manual calculation needed.
- **Booking overlap trigger**: The `trg_check_booking_no_overlap` trigger prevents double-booking on same date/listing. Synthetic data must respect this — seed script generates non-overlapping time slots.
- **Supabase pooler**: Use the same pooler connection string. No changes needed.
