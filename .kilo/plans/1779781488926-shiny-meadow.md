# Backend Development Plan: EventVenue.Asia

> v0.2.0 — MVP backend migration from localStorage mock data to production-ready API.

---

## 1. Technical Stack

| Layer | Choice | Justification |
|-------|--------|--------------|
| **Runtime** | Node.js 22 LTS (Next.js 15) | Already in project. Monolith in early stage fine. Can extract services later. |
| **Language** | TypeScript 5 | Type safety across full stack. Shared types between frontend/backend. |
| **Framework** | Next.js 15 App Router + API Routes | Zero-config API. Route grouping, middleware, streaming. Built-in edge runtime option. |
| **Database** | PostgreSQL 16 | Relational (venues, bookings, users). Strong GEO support (PostGIS) for location search. Row-level security for multi-tenancy. |
| **ORM** | Drizzle ORM | Lightweight, SQL-like API, excellent TypeScript inference. Better perf than Prisma for complex queries. Raw SQL escape hatch. |
| **Auth** | Auth.js v5 (NextAuth) + JWT | Multi-provider (Google, Apple, credentials). Session via JWT (no DB session table). Vendor + Admin role claims in token. |
| **File Storage** | Cloudflare R2 | S3-compatible, no egress fees. Cheap for image-heavy venue photos. Fronted by CDN. |
| **Search** | PostgreSQL Full-Text Search + pg_trgm | Built-in, no extra infra. Trigram similarity for fuzzy name search. tsvector for text search. Upgrade to MeiliSearch when scale demands. |
| **Caching** | Upstash Redis | Serverless Redis. Session cache, rate limiting, AI plan result caching. |
| **Email** | Resend | React email templates, good deliverability in SEA region. |
| **Real-time** | Pusher (Channels) | Simple WebSocket abstraction. For messaging + live inquiry notifications. |
| **AI** | OpenAI API (GPT-4o-mini) | Parsing natural language → structured venue query. Cost: ~$0.15/1K queries with structured output mode. |
| **Payments** | Stripe Connect | Marketplace split payments (venue owner + platform commission). Later phase. |
| **Deploy** | Vercel (app) + Supabase/Neon (DB) | Managed PostgreSQL. Serverless-friendly. No cold start issues with connection pooling. |
| **Monitoring** | Sentry + Axiom | Error tracking + structured logging. |

### Why not microservices?

MVP traffic <10K DAU. Monolith API routes under `/api/*` with service-layer separation (domain modules) is sufficient. Extract to separate services when:
- Messaging volume > 1K concurrent WS connections
- AI pipeline becomes bottleneck (move to worker/queue)
- Booking transaction volume > 100/sec

## 2. Data Structures & Schema Design

### 2.1 Entity Relationship Summary

```
User ──< Booking ──< BookingService
  │
  ├──< VendorProfile (1:1)
  │     ├──< VendorDocument (verification)
  │     └──< Listing (venues or services)
  │           ├──< ListingPhoto
  │           ├──< ListingAmenity (M:N)
  │           ├──< ListingEventType (M:N)
  │           ├──< ServicePackage (service listings only)
  │           ├──< AvailabilitySlot
  │           ├──< AvailabilityBlock
  │           └──< Review
  │
  ├──< Inquiry (customer → listing)
  ├──< Favorite (customer → listing)
  ├──< Conversation (M:N, users)
  │     └──< Message
  └──< PlanSession (AI smart planner)
        └──< PlanRecommendation
```

```sql
CREATE TYPE booking_status AS ENUM ('pending', 'confirmed', 'in_progress', 'completed', 'cancelled');
CREATE TYPE conversation_type AS ENUM ('direct', 'group');

-- Users
CREATE TABLE users (

CREATE TABLE vendor_documents (

  description TEXT,
  location VARCHAR(255),
  coordinates GEOGRAPHY(POINT, 4326),
  address TEXT,

  is_verified BOOLEAN NOT NULL DEFAULT false,
  average_rating DECIMAL(2,1) NOT NULL DEFAULT 0,

CREATE TABLE event_types (

);

-- Listing <-> EventType (M:N)
CREATE TABLE listing_event_types (

-- Service packages (service listings only)

  created_at TIMESTAMPTZ NOT NULL DEFAULT now()

-- Service tags
CREATE TABLE service_tags (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  listing_id UUID NOT NULL REFERENCES listings(id) ON DELETE CASCADE,
  tag VARCHAR(100) NOT NULL
);

-- Availability blocks (venue — per-day block)
CREATE TABLE availability_blocks (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  listing_id UUID NOT NULL REFERENCES listings(id) ON DELETE CASCADE,
  date DATE NOT NULL,
  is_blocked BOOLEAN NOT NULL DEFAULT false,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (listing_id, date)

-- Availability slots (venue — time-range appointments)
CREATE TABLE availability_slots (

  start_time TIMESTAMPTZ NOT NULL,

-- Inquiries
```

### 3.3 Constraints & Triggers

```sql
-- Denormalized rating update trigger
CREATE OR REPLACE FUNCTION update_listing_rating()
RETURNS TRIGGER AS $$
BEGIN
  UPDATE listings
  SET
    average_rating = (SELECT COALESCE(ROUND(AVG(rating)::numeric, 1), 0) FROM reviews WHERE listing_id = NEW.listing_id),
    review_count = (SELECT COUNT(*) FROM reviews WHERE listing_id = NEW.listing_id),
    updated_at = now()
  WHERE id = NEW.listing_id;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_update_listing_rating
AFTER INSERT OR UPDATE OR DELETE ON reviews
FOR EACH ROW EXECUTE FUNCTION update_listing_rating();

-- Prevent double-booking: no two bookings overlap on same date for same listing
CREATE OR REPLACE FUNCTION check_booking_no_overlap()
RETURNS TRIGGER AS $$
BEGIN
  IF EXISTS (
    SELECT 1 FROM bookings
    WHERE listing_id = NEW.listing_id
      AND event_date = NEW.event_date
      AND status NOT IN ('cancelled')
      AND id != COALESCE(NEW.id, '00000000-0000-0000-0000-000000000000'::uuid)
      AND (
        (NEW.start_time, NEW.end_time) OVERLAPS (start_time, end_time)
      )
  ) THEN
    RAISE EXCEPTION 'Booking time overlap on date % for listing %', NEW.event_date, NEW.listing_id;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_check_booking_no_overlap
BEFORE INSERT OR UPDATE ON bookings
FOR EACH ROW EXECUTE FUNCTION check_booking_no_overlap();

-- Auto-update updated_at
CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

DO $$
DECLARE
  t text;
BEGIN
  FOR t IN
    SELECT table_name FROM information_schema.columns
    WHERE column_name = 'updated_at' AND table_schema = 'public'
  LOOP
    EXECUTE format(
      'CREATE TRIGGER trg_update_%I BEFORE UPDATE ON %I FOR EACH ROW EXECUTE FUNCTION update_updated_at_column()',
      t, t
    );
  END LOOP;
END;
$$;
```

### 4.2 Authentication & Authorization

**JWT Token Design**

```json
// Access token payload
{
  "sub": "user-uuid",
  "email": "user@example.com",
  "role": "customer",
  "vendorId": null,
  "vendorType": null,
  "iat": 1716710400,
  "exp": 1716714000
}
```

**Auth Flow:**
1. Login/Signup → server returns `accessToken` (1h) + sets `refreshToken` in httpOnly cookie (7d).
2. All API calls: `Authorization: Bearer <accessToken>`.
3. Token expired → call `/auth/refresh` with cookie → new access token.
4. Sign-out → server clears refresh cookie + blacklists access token (Redis, TTL = remaining expiry).

**Middleware Authorization:**
- `authenticate()` — extracts & verifies JWT, sets `req.user`.
- `requireRole(...roles)` — checks `req.user.role`.
- `requireVendor(vendorId)` — checks listing/vendor ownership.

**Role Matrix:**

| Resource | Customer | Vendor | Admin |
|----------|----------|--------|-------|
| Browse listings | R | R | R |
| Own profile | CRU | CRU | CRU |
| Own inquiries | CRU | - | - |
| Vendor inquiries | - | RU | R |
| Own listings | - | CRUD | R |
| Any listing status | - | - | CRUD |
| Vendor approval | - | - | CRU |
| User suspension | - | - | U |
| Content moderation | - | - | U |

### 5.1 Customer Venue Discovery → Booking

### 5.3 AI Smart Planner Flow

```
Customer types natural language prompt:
"I need a halal wedding venue in KL for 200 guests next month
with catering and photography, budget RM15000"
    │
    ▼
POST /api/v1/ai/plan { prompt }
    │
    │  1. Create plan_session (status=processing)
    │  2. Push job to Redis queue (BullMQ or simple fetch)
    │  3. Return 202 { sessionId } immediately
    │
    ▼
Worker picks up job:
    │
    │  Step A — Parse prompt via OpenAI structured output
    │  ┌────────────────────────────────────────────────────┐
    │  │ System prompt: Extract structured params.          │
    │  │ Response schema: {                                 │
    │  │   eventType: string|null,                          │
    │  │   guestCount: number|null,                         │
    │  │   location: string|null,                           │
    │  │   budget: number|null,                             │
    │  │   halal: boolean|null,                             │
    │  │   services: string[] (catering|photography|dj|...) │
    │  │   confidence: number,                              │
    │  │   missingFields: string[]                          │
    │  │ }                                                  │
    │  └────────────────────────────────────────────────────┘
    │
    │  Step B — Build SQL query from parsed params
    │  ┌────────────────────────────────────────────────────┐
    │  │ SELECT ... FROM listings l                         │
    │  │ LEFT JOIN listing_amenities la ON ...              │
    │  │ LEFT JOIN listing_event_types let ON ...           │
    │  │ WHERE l.status = 'active'                          │
    │  │ AND l.listing_type = 'venue'                       │
    │  │ AND l.halal_certified = true                       │
    │  │ AND l.capacity >= 200                              │
    │  │ AND l.location ILIKE '%kuala lumpur%'             │
    │  │ ORDER BY l.average_rating DESC LIMIT 10            │
    │  └────────────────────────────────────────────────────┘
    │
    │  Step C — Score matches
    │  ┌────────────────────────────────────────────────────┐
    │  │ For each venue:                                    │
    │  │   score = capacity_match + location_match + ...    │
    │  │   estimatedCost = price_per_hour * 3h (assumed)    │
    │  │   matchReason = natural language explanation       │
    │  └────────────────────────────────────────────────────┘
    │
    │  Step D — Find matching service providers
    │  ┌────────────────────────────────────────────────────┐
    │  │ Query service listings where service_category IN   │
    │  │ ('catering', 'photography') AND location matches   │
    │  └────────────────────────────────────────────────────┘
    │
    │  Step E — Budget analysis
    │  ┌────────────────────────────────────────────────────┐
    │  │ totalEstimate = venueCost + sum(service costs)     │
    │  │ warnings if totalEstimate > budget                 │
    │  └────────────────────────────────────────────────────┘
    │
    │  Step F — Save results
    │  │   UPDATE plan_sessions SET status='completed', parsed_params = {...}
    │  │   INSERT plan_recommendations for each venue + service
    │  │   Cache in Redis (key: plan:{sessionId}, TTL: 1h)
    │
    ▼
Client polls GET /api/v1/ai/plan/:sessionId
    │  Returns cached result from Redis
    │  Or DB result if cache expired
    ▼
Customer sees UI results → can click into any venue/service via detail API
```

## Key Design Decisions & Rationale

1. **Drizzle over Prisma**: Drizzle has better raw SQL escape hatches for complex geo + FTS queries. Prisma's generated client is heavy for cold starts. Drizzle schema files double as TypeScript types.

2. **PostgreSQL over MongoDB**: Venue data is highly relational (listings → amenities, event types, photos, packages). FTS + PostGIS + constraint triggers (no double-booking) all native in PG.

3. **JWT (stateless) over DB sessions**: No session table join on every request. Refresh token in httpOnly cookie (secure, not JS-accessible). Token revocation via Redis blacklist (TTL = remaining expiry).

4. **Pusher over raw WebSocket**: No WebSocket server to manage. 200K free messages/day. Channel auth callbacks → no server-side state. Upgrade to Socket.io self-hosted when scale demands.

5. **AI Planner: async + polling, not streaming SSR**: MVP simplicity. OpenAI response ~2-5s, too long for a POST request timeout. Queue → poll pattern avoids Next.js function timeout (10s hobby, 60s pro).

6. **R2 over S3**: No egress fees. Venue-heavy images downloaded frequently by customers. S3 egress would be significant cost at scale.

7. **Monolith API routes**: MVP traffic manageable. Extract to separate services ONLY when: messaging > 1K concurrent WS, AI pipeline becomes bottleneck, booking transaction > 100/sec.
